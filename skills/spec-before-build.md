# Spec Before Build Skill

## Trigger

Use this before coding, especially when credits, time, or project complexity are constraints.

## Inputs

- Desired outcome.
- Target user.
- Existing workspace or repo.
- Known constraints.
- Definition of done.

## Process

1. Clarify the user-facing outcome.
2. Identify the smallest useful version.
3. List files or systems likely to change.
4. Define success checks.
5. Identify risks, unknowns, and dependencies.
6. Only then move into implementation.

## Output

A concise build spec with:

- Goal
- Non-goals
- Inputs
- Flow
- Files or systems
- Checks
- Next action
